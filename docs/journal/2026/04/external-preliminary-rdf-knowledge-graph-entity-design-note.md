Knowledge-Grounded Entity Design (Minimal First Approach)
========================================================

Status: draft
Author: ChatGPT
Date: 2026-04-11

---

# 1. Purpose

This note defines a minimal and evolutionary design approach for integrating
Entity (Fact World) with RDF / Context Graph / Embedding (Knowledge World)
in CML/CNCF.

The core principle is:

- Start with the minimal anchor (URI only)
- Discover additional knowledge attributes through actual use

---

# 2. Core Principle

Entity is the source of truth (fact world).
Knowledge is derived, contextual, and evolving.

Therefore:

- Do NOT over-design KnowledgeAttributes upfront
- Introduce only what is necessary when it becomes necessary

---

# 3. Minimal Design (Phase 1)

## Entity Definition

ENTITY Artwork

  id: EntityId
  title: String

  knowledge:
    uri: Uri

---

## Meaning

- `uri` is the RDF subject
- This establishes a direct identity link:

  Entity.id == RDF.subject

---

## Properties

This minimal design already enables:

- RDF graph integration
- Context Graph generation
- Embedding pipeline anchoring

---

# 4. Key Insight

> URI is sufficient for connection, but not for operation.

- URI solves identity
- Semantics, inference, and retrieval are layered on top

---

# 5. Separation of Concerns

## Fact World (Entity)

- authoritative state
- lifecycle management
- transactional integrity

## Knowledge World

- RDF: structural facts
- Context Graph: interpretation
- Embedding: similarity

---

## Relationship

Entity → RDF → Context Graph → Embedding

---

# 6. Design Philosophy

## 6.1 Avoid Premature Knowledge Modeling

Do NOT introduce:

- types
- contexts
- embedding
- control flags

until a real use case requires them.

---

## 6.2 Let Context Define Meaning

- type is not stored in Entity
- type is derived in Context Graph

Therefore:

  type = f(Entity, Context)

---

## 6.3 Keep Entity Clean

Entity should not:

- store inferred data
- store heavy data (embedding vectors)
- store context-dependent semantics

---

# 7. Evolution Strategy

KnowledgeAttributes evolve in phases:

---

## Phase 1 (Initial)

knowledge:
  uri

---

## Phase 2 (External Integration)

knowledge:
  uri
  same_as

Purpose:
- link to external knowledge graphs

---

## Phase 3 (RAG / Retrieval)

knowledge:
  uri
  same_as
  embedding_ref

Purpose:
- enable vector search

---

## Phase 4 (Context Optimization)

knowledge:
  uri
  same_as
  embedding_ref
  context_hints

Purpose:
- optimize context selection (optional)

---

## Important

- `types` is NOT required initially
- It may never be needed if type is fully context-derived

---

# 8. Design Rules

## Rule 1: Add only when needed

If there is no concrete use case, do not add attributes.

---

## Rule 2: Classify purpose

Every added attribute must belong to one category:

- Identity (uri, same_as)
- Retrieval (embedding_ref)
- Optimization (context hints)
- Control (rare)

---

## Rule 3: Do not store derived knowledge

Derived data belongs to:

- RDF
- Context Graph
- Embedding index

NOT Entity

---

## Rule 4: Keep references, not payloads

Correct:

  embedding_ref

Incorrect:

  embedding_vector

---

# 9. Role of Context Graph

Context Graph is responsible for:

- type assignment
- semantic interpretation
- view generation
- inference

Therefore:

  type ∈ ContextGraph

NOT:

  type ∈ Entity

---

# 10. Key Insight (Critical)

> Knowledge is not stored — it is generated by context.

---

# 11. Benefits of This Approach

- avoids over-engineering
- preserves domain purity
- supports incremental evolution
- aligns with RDF / inference model
- integrates naturally with RAG systems

---

# 12. Summary

Minimal viable integration:

  Entity + URI

Everything else:

  discovered through usage

---

# 13. Guiding Principle

> First connect. Then interpret. Then optimize.

---

# 14. One-line Philosophy

> Start with identity, grow into meaning.
